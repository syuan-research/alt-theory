# Decision: Electron Sidecar Architecture with Security Hardening

## Background
Alt Theory v0.3 is being packaged into an Electron desktop container to support deployment across multiple user groups. Alt Theory is expected to gain wider formal adoption by late 2026 or 2027. By then, AI coding agents are anticipated to be highly popular and widely used in social science research. 

Therefore, Alt Theory must serve two major user segments:
1. **Users without computer systems knowledge (non-programmers)**: They require a simple, single-click installable desktop GUI application.
2. **Users with systems knowledge (programmers/researchers)**: They already run coding agents (like Claude Code, Antigravity, etc.) and prefer a CLI plugin or skill integration format (e.g., oh-my-opencode-slim, Claude Code plugin).

We evaluated two paths for the desktop app wrapper: Branch A (running the Express/WebSocket backend as an embedded sidecar subprocess) and Branch B (refactoring all backend logic into Electron native IPC).

## Decision
We choose **Branch A (Sidecar Backend)** as our packaging architecture for Electron.
To mitigate the security and user experience vulnerabilities of standard sidecars (local port exposure, firewall alerts, and orphan processes), we will enforce the following hardening measures:
1. Bind the Express server strictly to the loopback interface (`127.0.0.1`).
2. Implement random Token-Based Authentication for both HTTP and WebSocket interfaces on startup.
3. Monitor `stdin` close and parent PID in the sidecar to prevent orphan processes.
4. Bundle a pre-compiled portable Node.js runtime inside the desktop package to guarantee execution on machines without Node.js.

## Rationale
1. **Target User Segments & Multi-Client Support**: Preserving the standard HTTP/WebSocket API allows Alt Theory to support both segments with a unified API design. For users with systems knowledge using coding agents, they can run the core headlessly or integrate it into CLI/Agent tools easily. At the same time, this same API is encapsulated inside the Electron shell for users without systems knowledge, ensuring zero dual-development overhead.
2. **Code Reuse & Maintainability**: Using a sidecar allows the exact same backend logic to be shared across web, desktop, and CLI plugins, maintaining a single codebase. Refactoring to Electron IPC would require double maintenance of parallel API layers.
3. **ABI Stability**: Standard Node.js execution avoids the fragile process of compiling native addons (like `@mariozechner/pi-ai`) against Electron's Node ABI via `electron-rebuild`.

## Alternatives Considered
- **Branch B (IPC Refactoring)**: Refactoring routers/listeners into Electron Main process IPC. While highly secure and sandboxed by default, it was rejected due to high refactoring overhead, duplication of the API layer, and lack of support for external CLI/Agent integrations.

## Consequences
- The application will require a pre-compiled Node.js binary bundled inside the assets, adding 30-50MB to the packaged application size.
- Backend scripts must be configured to unpack physically from the ASAR archive (`asar.unpack`) so the child process can access them.
- All frontend calls must include the authorization token generated by the Electron main process.

## Updates & Revisions (2026-06-02)
* **Rejection of the `ELECTRON_RUN_AS_NODE` Spawning Bypass**: During Stage 2 packaging verification, we verified a workaround that spawned the sidecar backend using the Electron executable itself configured with the `ELECTRON_RUN_AS_NODE=1` environment variable. This was designed to bypass a local system Node.js dependency. However, this has been rejected for production builds. Official Electron security hardened profiles recommend disabling the `runAsNode` fuse. Locking this fuse forces Electron to ignore `ELECTRON_RUN_AS_NODE=1`, which would crash the packaged application in production.
* **Resolution**: The temporary `ELECTRON_RUN_AS_NODE` spawn bypass is rolled back. Standard `npx tsx` spawning is restored for developer preview. Production builds must use a compiled JavaScript sidecar spawned via a standard Node execution pipeline or an embedded standalone Node.js sidecar binary.
