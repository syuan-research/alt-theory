  Alt Theory RAG System Design Review

  1. Overall Assessment

  This design is nearly ready for implementation with some critical clarifications needed. The biggest risk is the
  parent-child chunking implementation complexity — the design underestimates how deeply this will affect ChromaDB
  storage and retrieval logic. The provider abstraction is well-conceived, but the two-collection approach for
  parent-child isn't fully specified.

  Strengths: Config-driven architecture is excellent for a non-programmer user. The fork choice (knowledge-rag) is
  sound. The unified search framing (RAG + native grep) is pragmatic.

  Biggest risk: Parent-child chunking implementation details are underspecified, particularly the ChromaDB storage model
   and retrieval logic.

  ---
  2. Critical Issues (MUST fix before implementation)

  Issue #1: Parent-child ChromaDB storage model is underspecified

  What's wrong: The design says "parent-child chunking" but doesn't specify HOW this maps to ChromaDB collections. The
  current knowledge-rag uses a single flat collection. Parent-child requires either:
  - Two collections (parents + children) with metadata linking, OR
  - Single collection with chunk_type metadata filtering

  Why it matters: This is the core architectural decision. Getting it wrong means rebuilding the entire storage layer
  mid-implementation.

  Suggested fix:
  - Recommended approach: Single collection with chunk_type: "parent" | "child" metadata field
  - Add parent_id to child chunks (already in design doc Section 3.2)
  - Search hits children only (filter: chunk_type="child")
  - Retrieval: fetch child + query parent by parent_id to attach context
  - This is simpler than two collections and avoids cross-collection queries

  Code impact: Modify ingestion.py to create parent chunk (full document) + child chunks (sections). Modify server.py
  search to filter by chunk_type and post-process to attach parent context.

  ---
  Issue #2: Configurable separator regex won't parse v0.1 format correctly

  What's wrong: Design doc Section 3.2 proposes:
  separators:
    - "^===SECTION===$"

  But actual KB v0.1 files use ===SECTION=== as a line by itself, not necessarily at line start. The regex
  ^===SECTION===$ requires start-of-line + end-of-line with NOTHING else, which may fail if there's trailing whitespace.

  Why it matters: If the parser can't split v0.1 documents correctly, Day 1 testing fails.

  Suggested fix:
  separators:
    - "^===SECTION===\\s*$"  # Allow trailing whitespace

  Test this regex against actual v0.1 files before implementation. Better yet: write a small test script that parses one
   v0.1 file with the proposed regex and verifies section count.

  ---
  Issue #3: Metadata extraction from JSON frontmatter (v0.1) is not implemented

  What's wrong: Current ingestion.py only extracts YAML frontmatter (lines 197-202):
  frontmatter_match = re.match(r"^---\n(.*?)\n---\n", content, re.DOTALL)

  But v0.1 KB docs use JSON frontmatter (no --- delimiters, just { } at file start). The design doc mentions
  "configurable metadata extraction" but doesn't specify how to parse JSON frontmatter.

  Why it matters: Without this, v0.1 documents will have no metadata, breaking the entire metadata-driven search.

  Suggested fix: Add JSON frontmatter parser to ingestion.py:
  # Try JSON frontmatter first (v0.1 format)
  json_match = re.match(r"^\s*\{(.*?)\}\s*\n", content, re.DOTALL)
  if json_match:
      try:
          frontmatter_data = json.loads("{" + json_match.group(1) + "}")
          metadata.update(frontmatter_data)content = content[json_match.end():]  # Remove frontmatter
      except json.JSONDecodeError:
          pass
  # Then try YAML frontmatter (v0.2 format)
  elif content.startswith("---"):
      # existing YAML parser

  This allows the system to handle BOTH v0.1 (JSON) and v0.2 (YAML) formats without config changes.

  ---
  Issue #4: Provider abstraction Protocol is incomplete

  What's wrong: Design doc Section 3.1 shows:
  class EmbeddingProvider(Protocol):
      def embed(self, texts: List[str]) -> List[List[float]]: ...
      def dimension(self) -> int: ...

  But ChromaDB's embedding function interface requires __call__() method, not embed(). The current FastEmbedEmbeddings
  class (server.py lines 133-183) implements __call__().

  Why it matters: If the Protocol doesn't match ChromaDB's interface, provider swapping won't work.

  Suggested fix: Update Protocol to match ChromaDB interface:
  class EmbeddingProvider(Protocol):
      def __call__(self, input: List[str]) -> List[List[float]]: ...
      def dimension(self) -> int: ...

  Or keep embed() as the protocol method and have each provider implement both embed() and __call__() (where __call__
  delegates to embed).

  ---
  3. Important Concerns (SHOULD address, not blockers)

  Concern #1: Domain switching with active_domain.txt is fragile

  Potential impact: If the file gets corrupted or deleted, the system doesn't know which domain is active. Also,
  concurrent access (multiple Claude sessions) could cause race conditions.

  Suggested approach:
  - Store active domain in config.yaml instead (user-editable, version-controlled)
  - OR: Make domain an explicit parameter to MCP tools (no "active" state)
  - OR: Use a proper state file with locking (overkill for single-user)

  For single-user, simplest is: domain parameter in search tool, no global state.

  ---
  Concern #2: Query expansion config will need iteration

  Potential impact: The security-term expansions in knowledge-rag are extensive (lines 206-280 in config.py). Replacing
  these with environmental psychology terms is non-trivial. The design doc shows 5 example expansions (Section 3.6), but
   a real system needs 50+.

  Suggested approach:
  - Start with NO query expansion in Phase 1 (empty dict)
  - Add expansions incrementally based on retrieval quality testing
  - Don't block implementation on building a complete expansion dictionary

  ---
  Concern #3: Reranker model choice needs validation

  Potential impact: Design doc chooses bge-reranker-base (MIT license) over jina-reranker-v2 (CC-BY-NC-4.0). But does
  bge-reranker-base support Chinese? The design doesn't verify this.

  Suggested approach:
  - Test bge-reranker-base with Chinese queries before committing
  - If it fails, consider: (a) disable reranker for Phase 1, (b) use jina-reranker-v2 anyway (non-commercial use is
  allowed for personal projects), or (c) find another multilingual reranker

  ---
  Concern #4: BM25 index rebuild on every server start

  Potential impact: Current knowledge-rag rebuilds BM25 index from ChromaDB on startup (server.py lines 518-534). For
  <200 documents this is fast, but it's wasteful.

  Suggested approach:
  - Phase 1: Accept the rebuild (it's <1 second for 200 docs)
  - Phase 2: Serialize BM25 index to disk (pickle) and load on startup
  - Not a blocker, just inefficient

  ---
  4. Suggestions (nice-to-have)

  - Add a validation script: Before implementation, write a small script that tests the proposed regex separators and
  metadata extraction against actual v0.1 files. This catches parsing issues early.
  - Simplify MCP tools in Phase 1: The design says "keep all 12 tools initially" but this adds testing burden. Consider:
   keep only search_knowledge, reindex_documents, list_domains, switch_domain. Add others in Phase 2 after core works.
  - Add error handling for missing frontmatter: Some KB docs might have malformed or missing frontmatter. The parser
  should gracefully handle this (use filename as fallback for required fields like theory).
  - Consider a migration script: If v0.1 → v0.2 format transition is planned, a batch conversion script (Python) would
  be valuable. But this can be Phase 3.

  5. Answers to Open Questions (from design doc Section 7)

  Q1: Unified search architecture details — How should agent.md encode search strategy?

  Recommended approach for agent.md:

  ## Knowledge Retrieval Strategy

  Use progressive disclosure based on query type:

  **Level 0 - Theory List** (when user asks "what theories exist?")
  - Use native grep: `grep -r "theoryname:" documents-env-psychology/`
  - Fast, no RAG needed

  **Level 1 - Section Titles** (when user asks "what does ART cover?")
  - Read the core document directly (if you know the filename)
  - OR use RAG with top_k=1 to find the document, then Read it

  **Level 2 - Semantic Retrieval** (when user asks conceptual questions)
  - Use MCP tool `search_knowledge` with query
  - RAG returns relevant child chunks with parent context

  **Level 3 - PDF Full Text** (when user needs original source)
  - Check sources.yaml for PDF path
  - Use Read tool on the PDF file

  This gives the agent clear decision rules. The key insight: grep for known-item search, RAG for semantic search.

  ---
  Q2: Does Claude Code native search actually beat RAG?

  Honest answer: For <200 markdown documents, native grep beats RAG for:
  - Exact keyword lookup (theory names, author names)
  - Known-item search ("find the ART document")
  - Browsing section titles

  RAG adds value for:
  - Semantic similarity: "theories about mental fatigue" → finds ART even if query doesn't say "attention restoration"
  - Hybrid ranking: BM25 + vector fusion surfaces better results than pure keyword
  - Parent-child retrieval: Grep can't automatically attach parent context to section matches

  Verdict: RAG is worth it, but ONLY with parent-child chunking. Without parent-child, grep + Read is simpler and nearly
   as good.

  ---
  Q3: PDF source mapping format — Confirm sources.yaml?

  Answer: Yes, sources.yaml is the right approach. Keep it separate from KB docs (not version-controlled if PDFs are
  copyrighted). Format is good as proposed.

  Alternative: Embed PDF paths in frontmatter with a pdf_path field, but mark the field as "local only, do not share".
  But separate file is cleaner.

  ---
  Q4: Initial embedding model choice — multilingual-e5-large vs smaller?

  Answer: Start with paraphrase-multilingual-MiniLM (384D, ~420MB) instead of multilingual-e5-large (1024D, 2.24GB).

  Rationale:
  - User has limited disk/memory
  - 384D is sufficient for <200 documents (not a large-scale system)
  - FastEmbed supports both models
  - Provider abstraction allows easy upgrade to e5-large or BGE-M3 later

  Test with MiniLM first. If retrieval quality is poor, upgrade to e5-large. Don't over-engineer for scale you don't
  have yet.
