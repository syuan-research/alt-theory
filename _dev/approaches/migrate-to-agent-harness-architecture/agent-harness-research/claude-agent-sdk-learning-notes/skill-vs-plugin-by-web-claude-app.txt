## 1. Skill vs Plugin Architecture

These are fundamentally different abstractions serving different purposes.

**Skills** are *context injectors* — passive knowledge documents. Rather than performing actions directly, they inject specialized instructions into the conversation history and dynamically modify Claude's execution environment. Skills modify conversation context by injecting instruction prompts and modify execution context by changing tool permissions and model selection. Skills cannot define hooks on their own — they are purely about *what Claude knows and how it reasons* when activated.

**Plugins** are *delivery vehicles* that bundle multiple primitives together. A plugin can include skills, agents, hooks, MCP servers, and LSP servers. The plugin itself is just packaging; the interesting primitives live inside it.

**What a Plugin can do that a Skill cannot:**

- **Define hooks** — automated event-driven scripts that run regardless of Claude's in-context reasoning
- **Bundle MCP servers** — wire in external tool integrations
- **Include agents** — define specialized subagent workflows
- **Ship slash commands** — user-invocable `/my-plugin:command` workflows

The key distinction: use a skill when you want Claude to load richer instructions only when they are relevant, or when you want a named workflow you can invoke directly. Use a hook when the system should enforce or inject something at a specific lifecycle event. Hooks are not a softer version of skills — they are automation points.

### Hook Events Reference

The supported hook events are:

| Event | When | Use For |
|---|---|---|
| `PreToolUse` | Before tool runs | Validation, modification |
| `PostToolUse` | After tool completes | Feedback, logging |
| `UserPromptSubmit` | User submits prompt | Context injection, validation |
| `Stop` | Main agent considers stopping | Completeness check |
| `SubagentStop` | Subagent considers stopping | Task validation |
| `SessionStart` | Session begins | Context loading |
| `SessionEnd` | Session ends | Cleanup, logging |
| `PreCompact` | Before context compaction | Preserve critical context |
| `Notification` | Claude sends notification | Logging, reactions |

There are two hook types: **prompt-based hooks** (recommended) use LLM-driven decision making for context-aware validation and are supported on `Stop`, `SubagentStop`, `UserPromptSubmit`, and `PreToolUse`. **Command hooks** execute bash scripts for deterministic checks — use these for fast validations, file system operations, or external tool integrations.

## 2. Dynamic Behavior with Plugins — Your User Personalization Use Case

### Can hooks read external files?

Yes. Command hooks have access to environment variables including `$CLAUDE_PROJECT_DIR` (project root), `$CLAUDE_PLUGIN_ROOT` (plugin directory), and `$CLAUDE_ENV_FILE` (SessionStart only — persist env vars here). A `SessionStart` command hook can trivially `cat user_profile.json` and inject its contents.

### Can hooks maintain state across turns?

Not natively — hooks don't have a built-in state store. The recommended pattern is to use the filesystem: write state to a file in `$CLAUDE_PROJECT_DIR/.claude/` between turns, and read it back in subsequent hook invocations. The `$CLAUDE_ENV_FILE` mechanism in `SessionStart` also lets you persist environment variables for the session lifetime.

### Implementing "every N turns, inject X context"

The `UserPromptSubmit` hook fires on every user message, making it perfect for this. Here's the pattern using a command hook:

```bash
#!/bin/bash
# hooks/inject-profile.sh
COUNTER_FILE="$CLAUDE_PROJECT_DIR/.claude/turn_counter"
PROFILE_FILE="$CLAUDE_PROJECT_DIR/user_profile.json"
N=5  # inject every 5 turns

# Read and increment counter
count=$(cat "$COUNTER_FILE" 2>/dev/null || echo 0)
count=$((count + 1))
echo "$count" > "$COUNTER_FILE"

# Every N turns, inject profile context
if (( count % N == 0 )) && [ -f "$PROFILE_FILE" ]; then
  profile=$(cat "$PROFILE_FILE")
  echo "{\"systemMessage\": \"User profile context: $profile\"}"
else
  echo '{"continue": true}'
fi
```

With hook config:

```json
{
  "UserPromptSubmit": [{
    "matcher": "*",
    "hooks": [{
      "type": "command",
      "command": "bash ${CLAUDE_PLUGIN_ROOT}/hooks/inject-profile.sh",
      "timeout": 5
    }]
  }]
}
```

### Recommended pattern for user personalization

The cleanest architecture combines two hooks:

**`SessionStart`** — load the full profile once at session open and set environment variables:
```bash
profile=$(cat "$CLAUDE_PROJECT_DIR/user_profile.json" | jq -r '.expertise_level')
echo "export USER_EXPERTISE=$profile" >> "$CLAUDE_ENV_FILE"
echo '{"systemMessage": "Loaded user profile. Adapt your responses accordingly."}'
```

**`UserPromptSubmit` (prompt-based)** — dynamically adjust tone per message:
```json
{
  "UserPromptSubmit": [{
    "matcher": "*",
    "hooks": [{
      "type": "prompt",
      "prompt": "The user's expertise is $USER_EXPERTISE. If this prompt involves technical depth, adjust your planned response complexity accordingly."
    }]
  }]
}
```

One important caveat: hooks are loaded when Claude Code session starts — changes to hook configuration require restarting Claude Code. Editing `hooks/hooks.json` won't affect the current session. So profile data can be dynamic (read from files at hook runtime), but the hook logic itself is fixed per session.
