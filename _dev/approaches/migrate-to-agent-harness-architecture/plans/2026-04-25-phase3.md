- [ ] **Step 3: 创建 WebSocket 协议定义文件**

```typescript
// websocket-protocol.ts
// 前后端共享的协议类型

export interface SessionSnapshot {
  sessionId: string;
  status: "idle" | "running" | "error";
  currentDomain: string;
  profilePath?: string;
  messageCount: number;
}

export type ClientMessage =
  | { type: "prompt"; payload: string }
  | { type: "abort" }
  | { type: "switch_kb"; payload: { domain: string } }
  | { type: "switch_profile"; payload: { profilePath: string } }
  | { type: "new_session" };

export type ServerMessage =
  | { type: "session_opened"; payload: SessionSnapshot }
  | { type: "session_updated"; payload: SessionSnapshot }
  | { type: "assistant_delta"; payload: { text: string } }
  | { type: "tool_started"; payload: { toolName: string; callId: string } }
  | { type: "tool_updated"; payload: { callId: string; text?: string; progress?: number } }
  | { type: "tool_finished"; payload: { callId: string; success: boolean; output?: unknown } }
  | { type: "run_completed"; payload: SessionSnapshot }
  | { type: "run_failed"; payload: { error: string } }
  | { type: "error"; payload: { error: string } };
```

**注意**：`tool_started` 不传递 `input`（原始命令），前端只显示人类可读文案。

- [ ] **Step 4: 创建 WebSocket 服务器**

核心逻辑：
1. Express 提供静态文件服务（pi-web-ui 前端），**共享 HTTP server**（不用单独 3001 端口）
2. WebSocket 处理实时通信
3. 每个 WebSocket 连接对应一个 PI `AgentSession`
4. PI SDK 事件 → WebSocket 消息（参考 spec v0.6 §7.3 事件映射）

```typescript
// server.ts 框架
import express from "express";
import { createServer } from "http";
import { WebSocketServer } from "ws";
import { createAltTheorySession } from "../core/alt-theory-core.js";
import type { ClientMessage, ServerMessage, SessionSnapshot } from "./websocket-protocol.js";

const app = express();
const server = createServer(app);
const wss = new WebSocketServer({ server });

// Server state（集中管理，不要散落）
const serverState = {
  currentDomain: "ep-core",
  currentProfilePath: "./assets/profiles/default.md",
};

wss.on("connection", async (ws) => {
  const { session } = await createAltTheorySession({
    rootDir: process.cwd(),
    kbDir: "./assets/kb",
    readOnly: true,
  });

  // 发送 session_opened
  const snapshot: SessionSnapshot = {
    sessionId: session.sessionId ?? "unknown",
    status: "idle",
    currentDomain: serverState.currentDomain,
    profilePath: serverState.currentProfilePath,
    messageCount: 0,
  };
  ws.send(JSON.stringify({ type: "session_opened", payload: snapshot }));

  // PI SDK 事件 → WebSocket
  // ⚠️ 注意：必须用 Step 2 验证过的实际 API（subscribe / on）
  session.subscribe((event) => {  // 如果实际是 .on()，改成 .on()
    switch (event.type) {
      case "message_update":
        if (event.assistantMessageEvent?.type === "text_delta") {
          const msg: ServerMessage = {
            type: "assistant_delta",
            payload: { text: event.assistantMessageEvent.delta ?? "" }
          };
          ws.send(JSON.stringify(msg));
        }
        break;
      case "tool_execution_start":
        const msg: ServerMessage = {
          type: "tool_started",
          payload: { toolName: event.toolName, callId: event.toolCallId }
        };
        ws.send(JSON.stringify(msg));
        break;
      // ... 其他事件映射（参考 spec v0.6 §7.3）
    }
  });

  // WebSocket → PI SDK
  ws.on("message", (data) => {
    const msg = JSON.parse(data.toString()) as ClientMessage;
    switch (msg.type) {
      case "prompt": {
        const contextPrefix = serverState.currentDomain !== "all"
          ? `[Context: Search in ./assets/kb/${serverState.currentDomain}/ unless user says otherwise.]\n`
          : "";
        // ⚠️ prompt() 可能是 async，用 .catch() 捕获错误
        session.prompt(contextPrefix + msg.payload).catch((err: Error) => {
          ws.send(JSON.stringify({
            type: "run_failed",
            payload: { error: err.message }
          }));
        });
        break;
      }
      case "abort":
        // ⚠️ 方法名需按 Step 2 验证结果调整（abort / cancel / stop）
        session.abort();
        break;
      case "switch_kb":
        serverState.currentDomain = msg.payload.domain;
        break;
      case "switch_profile":
        serverState.currentProfilePath = msg.payload.profilePath;
        break;
    }
  });
});

// 共享 HTTP server，统一端口
server.listen(3000, () => {
  console.log("Server running on http://localhost:3000");
});
```

**注意**：`session.subscribe()`、`session.prompt()`、`session.abort()` 的方法名必须按 Step 2 验证结果调整。如果 `AgentSession` 没有 `subscribe()`，需要用 `on()` 或其他 API。

- [ ] **Step 1: 写前端代码（vanilla HTML + 手动渲染）**

**决策：不用 pi-web-ui 组件。**

原因：`createAltTheorySession()` 返回 `AgentSession`（来自 `pi-coding-agent`），而 `pi-web-ui` 期望 `Agent`（来自 `pi-agent-core`）。这两个是不同的类，API 不兼容。Agent facade 方案大概率会失败。

用 vanilla HTML + JS 手动渲染 WebSocket 消息：


```javascript
// client.js
const ws = new WebSocket("ws://localhost:3000");  // 统一端口 3000

const messagesEl = document.getElementById("messages");
const inputEl = document.getElementById("input");
const sendBtn = document.getElementById("send");
const stopBtn = document.getElementById("stop");
const toolStatusEl = document.getElementById("tool-status");

let currentAssistantMessage = null;

// WebSocket 消息 → UI
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  switch (msg.type) {
    case "session_opened":
      console.log("Session opened:", msg.payload.sessionId);
      break;
    case "assistant_delta": {
      if (!currentAssistantMessage) {
        currentAssistantMessage = document.createElement("div");
        currentAssistantMessage.className = "message assistant";
        messagesEl.appendChild(currentAssistantMessage);
      }
      currentAssistantMessage.textContent += msg.payload.text;
      break;
    }
    case "tool_started": {
      const toolText = {
        "read": "正在读取知识库...",
        "grep": "正在搜索相关理论...",
        "find": "正在查找文件...",
        "ls": "正在列出文件...",
      }[msg.payload.toolName] || "正在处理...";
      toolStatusEl.textContent = toolText;
      break;
    }
    case "tool_finished": {
      toolStatusEl.textContent = "✓ 完成";
      setTimeout(() => { toolStatusEl.textContent = ""; }, 2000);
      break;
    }
    case "run_completed": {
      currentAssistantMessage = null;
      sendBtn.style.display = "inline";
      stopBtn.style.display = "none";
      break;
    }
    case "run_failed": {
      const errEl = document.createElement("div");
      errEl.className = "message error";
      errEl.textContent = "Error: " + msg.payload.error;
      messagesEl.appendChild(errEl);
      currentAssistantMessage = null;
      break;
    }
  }
  messagesEl.scrollTop = messagesEl.scrollHeight;
};

// 发送消息
sendBtn.onclick = () => {
  const text = inputEl.value.trim();
  if (!text) return;

  const userEl = document.createElement("div");
  userEl.className = "message user";
  userEl.textContent = text;
  messagesEl.appendChild(userEl);

  ws.send(JSON.stringify({ type: "prompt", payload: text }));
  inputEl.value = "";
  sendBtn.style.display = "none";
  stopBtn.style.display = "inline";
};

// 停止生成
stopBtn.onclick = () => {
  ws.send(JSON.stringify({ type: "abort" }));
};

// KB 切换
document.getElementById("kb-select").addEventListener("change", (e) => {
  ws.send(JSON.stringify({ type: "switch_kb", payload: { domain: e.target.value } }));
});

// Profile 切换
document.getElementById("profile-select").addEventListener("change", (e) => {
  ws.send(JSON.stringify({ type: "switch_profile", payload: { profilePath: e.target.value } }));
});
```

**注意**：这是 MVP 阶段最简实现。未来如果需要 pi-web-ui 的高级功能（artifacts、attachments），需要重新评估接入方案。

**已包含工具调用视觉处理**：
- `tool_started` → 人类可读文案映射（"正在搜索相关理论..."等）
- `tool_finished` → "✓ 完成"，2 秒后消失
- 不显示原始命令细节（如 `grep -r "ART" ./kb`）

以上逻辑已在 client.js 的 `ws.onmessage` switch 中实现，无需额外步骤。

## 已知问题（不阻塞 MVP）

| 问题 | 说明 |
|------|------|
| PowerShell | PI bash 工具只支持 bash，Windows 用户需要 Git for Windows |
| Prompt 占位符 | v0.6 prompt 有 `<context>[Insert retrieved KB results]</context>`，需要用户更新 |
| KB 内容为空 | `kb/` 目录无文件，数据层问题 |
| 隐藏消息注入 | 当前用文本前缀拼接（用户可见），未来可改用 PI `steer()` 或 `display: false` |
| switch_profile 无实际效果 | 切换 profile 只更新 `serverState`，当前 session 的 system prompt 不会变。下一条消息不会注入新 profile。**UI 需加提示："切换用户档案将在下次新对话生效"**。原因是 profile 影响整体行为模式，不适合 context prefix 注入 |

## 后续路线图提醒（Opus Review 洞察）

> 以下问题不在 Phase 3 MVP 实现，但记录在此作为"提醒"。
> 来源：`opus-deep-review-phase3-plan.md` P2/P3 部分。
> 这些洞察的价值不在于当下一定需要，而在于：如果没有提醒，后续可能也不会考虑。

### R1: EventMapper 抽象层（P2）

当前 plan 把 PI SDK 事件映射直接写在 `server.ts` 里。随着事件增多（`compaction_start/end`、`turn_start/end`），`server.ts` 会膨胀。

**未来方案**：抽出 `event-mapper.ts`，职责单一——PI SDK event → ServerMessage。
`server.ts` 只做 WebSocket 管理 + session 生命周期。

**触发条件**：`server.ts` 事件映射 switch 超过 10 个 case 时，或需要加第三种事件源时。

### R2: ServerState 正式化（P2）

当前用裸对象 `serverState = { currentDomain, currentProfilePath }`。MVP 够用。

**未来方案**：用类或接口封装，集中管理所有运行时状态（session、domain、profile、messageHistory）。

**触发条件**：状态字段超过 5 个，或需要在多处读写状态时。

### R3: 多 session 预留口（P3）

当前单 session。未来多 session（树状分支）。

**最小预留**（现在就可以加，零成本）：
- `ClientMessage` 加可选 `sessionId?: string`，MVP 阶段忽略
- `session_opened` 的 payload 已含 `sessionId`（Task 1 Step 3 已定义），前端存起来备用

### R4: Profile 多条拼装（P3）

当前 `AltTheoryConfig.profilePath?: string`（单条）。

**未来**：`profilePaths?: string[]`，用户勾选多条 profile 条目，拼装后注入。

**最小预留**：在 `AltTheoryConfig` 注释中标注未来字段。

### R5: pi-web-ui 重新评估（P3）

当前不用 pi-web-ui 组件（`AgentSession` ≠ `Agent` 类型不兼容）。

**未来重新评估条件**：
- PI SDK 统一 `AgentSession` 和 `Agent` 的 API
- 或需要 artifacts、attachments 等高级功能
- 或 pi-web-ui 提供独立的 WebSocket 模式（不需要 Agent 实例）

### R6: 隐藏注入实现（P3）

当前 KB 切换的 context 注入是文本前缀拼接，用户复制消息时可见 `[Context: Search in ./assets/kb/urban/...]`。

**未来方案**：用 PI `steer()` 方法或 `before_agent_start` 的 message 注入（`display: false`），实现真正的隐藏注入。

**触发条件**：用户反馈 context 前缀干扰阅读体验时。
